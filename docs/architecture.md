# Architecture

Project Limbo is built around a small set of stable contracts. The runtime is
stdlib-only and organized as focused modules under `src/limbo/`, each owning one
concern. This document describes how they fit together; the [README](../README.md)
documents each subsystem's user-facing surface, and this file cross-links to
those sections.

## Module map

| Module | Responsibility |
|---|---|
| `spec.py` | Parse and validate the pipeline JSON into a `PipelineSpec` of `TaskSpec`s (with retry policy, sandbox reference, and a pipeline-level `Policy`). |
| `graph.py` | Deterministic topological planning; compute downstream/blocked sets. |
| `retry.py` | `RetryPolicy` — backoff strategies and retryable-failure classification. |
| `policy.py` | `Policy` — command allow/deny lists, env inheritance, sandbox profiles. |
| `engine.py` | `LocalExecutor` — scheduling, retries, cache, artifacts, and event emission. |
| `operators.py`, `expressions.py` | Built-in JSONL/CSV data operators and a sandboxed expression evaluator. |
| `cache.py`, `fingerprint.py` | Content-fingerprint cache and input/output digesting. |
| `leases.py` | `LeaseStore` — the worker lease protocol for local and remote workers. |
| `artifacts.py` | Content-addressed artifact store and `Artifact` metadata. |
| `observability.py` | Lifecycle event log, run metrics, and secret redaction. |
| `cli.py` | The `limbo` command surface. |

## Pipeline Spec

The pipeline specification is JSON so it can be generated by any language and
parsed without optional dependencies. The `version` field is required and gives
the project room to introduce richer schemas without silently changing behavior.
Beyond `command`/`operator` tasks, a spec may carry per-task `retry` and
`sandbox` fields and a top-level `policy` block (see
[Policy & Safety](#policy--safety)).

## Planner

The planner (`graph.py`) validates the task graph, rejects unsafe shapes
(duplicate IDs, missing dependencies, cycles), and produces deterministic
topological levels. Tasks in the same level can run in parallel because all of
their dependencies have already completed.

## Executor

The local executor (`engine.py`) maintains a ready queue and a thread pool. A
task is scheduled when all dependencies are complete; the effective environment
for a command is resolved through the pipeline's environment policy rather than a
blanket copy of `os.environ`. If a task fails, downstream tasks are marked
`blocked`. Every run writes a manifest (statuses, durations, return codes, log
paths, per-task attempt history and artifacts, and aggregate metrics) and a
JSONL event stream for post-run inspection.

Task lifecycle states are shared by the local executor and the lease protocol:
`pending`, `running`, `succeeded`, `failed`, `blocked`, `skipped`.

## Retry & Resume

`retry.py` defines a `RetryPolicy` (fixed/linear/exponential backoff, an optional
delay cap, retryable exit codes, and retryable-timeout handling). The executor
runs an attempt loop and records each attempt in the manifest. Retry
configuration is deliberately **excluded from the fingerprint** so the cache
stays deterministic across retries and resumes. `limbo resume <run-id>` reloads
the prior run's pipeline from its manifest and re-executes only failed, blocked,
or never-run tasks — succeeded work is carried forward through the cache. See the
README [Reliability](../README.md#reliability-retries-and-resumable-runs) section.

## Cache

The cache (`cache.py`) is a JSON file keyed by task ID. Each successful task
records its fingerprint, run ID, and (when an artifact store is configured) the
digests of its outputs. A cache hit requires a matching fingerprint, a previous
success, and outputs still present — and, when output digests were recorded,
that each output still matches its digest (so a silently edited or corrupted
output invalidates the cache). Fingerprints (`fingerprint.py`) include task
configuration and declared input content digests, making cache behavior explicit
and predictable.

## Worker Lease Protocol

`leases.py` extracts scheduling state into a `LeaseStore` so the same pipeline can
run under one local worker or many remote ones. Workers interact through
`claim → heartbeat/renew → complete/fail`:

- only tasks whose dependencies have completed are ever claimable (a worker can
  never start a task before its inputs exist);
- each claim mints a fresh **HMAC-signed** lease token that authenticates every
  follow-up call — tampered or forged tokens are rejected;
- leases **expire** after a timeout without a heartbeat and become claimable
  again, and the previous holder is fenced out (its token stops working), so a
  crashed worker cannot strand a task;
- a failed task blocks all downstream dependents.

`run_workers()` drives one worker (single-process mode) or many concurrent
workers through the same protocol. This realizes the "future distributed runtime"
the earlier local executor was shaped for. See the README
[Worker Lease Protocol](../README.md#worker-lease-protocol) section.

## Artifact Store

`artifacts.py` provides a content-addressed blob store: every artifact is
addressed by the SHA-256 digest of its contents, which makes storing the same
bytes twice idempotent and makes on-disk corruption detectable by recomputing the
hash. An `Artifact` records the digest, size, media type, producing task, and
logical path. When a store is configured, the executor ingests each succeeded
task's outputs, references them in the run manifest, and upgrades cache
validation from "outputs exist" to "outputs match their recorded digest." See the
README [Artifact Store](../README.md#artifact-store) section.

## Observability

`observability.py` makes runs debuggable. Every run writes a JSONL **event
stream** (`.limbo/runs/<run-id>/events.jsonl`) capturing task lifecycle
transitions, and records **metrics** (task counts, cache hits, failure/blocked
counts, run/queue time) in the manifest. Environment metadata and Limbo-generated
messages are passed through a **redactor** so secret-looking values (by name or
token shape) never reach disk. `limbo inspect` and `limbo timeline` read these
back. See the README [Observability](../README.md#observability) section.

## Policy & Safety

`policy.py` constrains how a pipeline's tasks run — useful for untrusted or
semi-trusted definitions:

- **command allow/deny lists** (deny wins; a set allowlist is fail-closed — a
  denied command is never executed and the run exits non-zero);
- **environment inheritance** (all / none / an explicit allowlist of names);
- **sandbox profiles**, a validated declarative model referenced per task, whose
  enforcement is left to future platform-specific backends.

See the README [Policy & Safety](../README.md#policy--safety) section.

## Run Metadata

Each run receives a unique ID and writes per-task stdout/stderr under
`.limbo/runs/<run-id>/<task-id>/`, alongside `manifest.json` (the structured
summary) and `events.jsonl` (the lifecycle stream). `limbo runs` lists past runs
newest-first so a `<run-id>` is easy to find for `inspect`, `timeline`, and
`resume`.

## Toward a Distributed Scheduler

With the lease protocol, artifact store, observability, and policy foundations in
place, the remaining architecture step (tracked as issue #8) is a durable
scheduler service: persisting pipeline/task/lease/worker/artifact state, fair
queue selection, backpressure, and reconnecting workers — while preserving the
local executor as the single-process path through the same contracts.
