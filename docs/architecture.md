# Architecture

Project Limbo is built around a small set of stable contracts.

## Pipeline Spec

The pipeline specification is JSON so it can be generated by any language and parsed without optional dependencies. The `version` field is required and gives the project room to introduce richer schemas without silently changing behavior.

## Planner

The planner validates the task graph, rejects unsafe shapes, and produces deterministic topological levels. Tasks in the same level can run in parallel because all of their dependencies have already completed.

## Executor

The local executor maintains a ready queue and a pool of workers. A task is scheduled when all dependencies are complete. If a task fails, downstream tasks are marked as blocked, and the executor writes a manifest for post-run inspection.

## Cache

The cache is intentionally simple: a JSON file keyed by task ID. Each successful task records the latest fingerprint and run ID. A cache hit requires:

- matching task fingerprint,
- previous success,
- all declared outputs still present.

Fingerprints include task configuration and declared input content digests. This makes cache behavior explicit and predictable.

## Run Metadata

Each run receives a unique ID and writes per-task stdout/stderr files under `.limbo/runs/<run-id>/<task-id>/`. A manifest summarizes statuses, durations, return codes, and log paths.

## Future Distributed Runtime

The current local executor is deliberately shaped like a scheduler core. Future remote workers can reuse the same task states:

- pending,
- running,
- succeeded,
- failed,
- blocked,
- skipped.

The next architecture step is to extract task leasing from local thread scheduling so remote workers can claim runnable tasks with heartbeats and lease expiry.
